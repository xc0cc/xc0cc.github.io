AI应用的一些发展

AI 是怎么一步步”长脑子”的：从 Transformer 到 Harness Engineering

参考说明：本文的时间线框架参考了《从 LLM 到 Agent》飞书文档中整理的 AI 发展脉络，结合自己的记忆和理解重新描述。感谢原作者的梳理工作。

最近几年 AI 发展太快了，新概念一个接一个往外蹦——Prompt Engineering、RAG、Agent、MCP、Agent Skill……单个看好像都懂，但串起来是什么关系？

这篇文章把 AI 从 GPT-3.5 爆火到现在的演进脉络串一遍。12 个节点，每个都在解决上一个没搞定的问题。

Key Takeaways

  • Transformer 的 Decoder-Only 路线是今天所有主流大模型的底层骨架
  • GPT-3.5 时代，一套好 Prompt 能卖钱，吴恩达的免费课被转疯
  • LoRA 先在 Stable Diffusion 绘画圈火起来，Civitai 上至今有上万个社区 LoRA，后来才普及到 LLM
  • Agent = Prompt + LLM + Tools，最简 Agent 几十行代码就能跑
  • 离职的同事终将化作温暖的 Skill——Agent Skill 让 SOP 可沉淀、可复用
  • Harness Engineering 是最后一环：不是让 AI 更强，而是给它套上缰绳

1. LLM——Transformer 铺的路

2017 年 Google 发了篇论文叫 《Attention Is All You Need》，提出了 Transformer 架构。核心想法是”注意力机制”——模型处理每个词的时候，不再只看前后几个词，而是直接关注整句话里所有跟它相关的词。并行计算、效率高、长文本也 hold 住。

GPT 系列选了 Decoder-Only（只用解码器）这条路线，专注做一件事：给定上文，预测下一个词。这个选择决定了现在 ChatGPT、Claude 这些主流模型的底层骨架。

关于 Transformer 的底层细节这里不展开了。只需要知道：现在所有主流大模型，底层都是 Decoder-Only 的 Transformer 变体。

2. Prompt Engineering——GPT-3.5 时代最火的技能

ChatGPT（GPT-3.5）爆火那阵，Prompt Engineering 是第一个出圈的概念。

原因很简单：同样的模型，问法不同，答案天差地别。问”介绍一下法国大革命”，跟问”用 200 字以内概括法国大革命的起因、经过、结果”，产出的质量完全不是一个级别。

那段时间特别疯狂——一套好的提示词模板是可以卖钱的。社交媒体上到处是”这 10 个 Prompt 让你效率翻倍”的帖子，PromptBase 这类交易平台也冒了出来。吴恩达（Andrew Ng）还跟 OpenAI 合作出了门免费的 Prompt Engineering 课程，那门课在当时被转疯了。

Prompt 分两种：系统提示词设定人设和规则，全程生效；用户提示词是每条具体问题的输入。

核心洞察很简单但很深刻：不用改模型参数，只靠优化输入就能显著提升效果。成本极低，ROI 极高。

不过热潮过后也冷静了——光靠 Prompt 有上限，复杂场景还是不够稳定。所以才有了后面的 Fine-tuning。

3. Fine-tuning——Prompt 搞不定了，动参数

Prompt 再强，碰到特定领域术语、固定格式输出、某种风格的一致性，光靠提示词很难稳定复现。那就得动模型本身。

**微调（Fine-tuning）**就是在预训练好的大模型基础上，用特定数据再训一轮，让模型对这个任务更擅长。

但全量微调太贵，大模型参数动辄几百 B。**LoRA（Low-Rank Adaptation）**解决了这个问题：不碰原始参数，只在旁边挂几个低秩矩阵来训练，参数量少好几个数量级，效果却不差。

LoRA 其实先在 Stable Diffusion 绘画圈火起来的——用几张照片训一个 LoRA，就能生成各种风格的自画像；用某个画风的作品集训一个 LoRA，Stable Diffusion 就能稳定出那种风格的图。C 站（Civitai）上至今有成千上万个社区训好的 LoRA 可下载。后来 LoRA 才在 LLM 领域普及，成为微调的主流方案。

Prompt 是让模型用现有智力解新题，Fine-tuning 是给模型补课。

4. RAG——模型记性不好，配个”外挂知识库”

大模型有个致命问题：训练数据有截止日期。问”昨天发生了什么”不知道，而且训练数据里的错误会被模型一本正经地复述出来。

RAG（Retrieval-Augmented Generation）的思路很直接：先查资料，再回答问题。

流程：用户提问 → 去外部知识库检索相关内容 → 把检索结果拼到 Prompt 里 → 模型基于这些材料生成回答。模型不再全凭”记忆”，而是拿着”参考资料”说话。准确性、时效性两个痛点同时解决。

5. Function Calling——光会说不够，得会做事

前面的模型再强，也只能输出文字。现实世界需要模型查数据库、发邮件、调日历……光靠文本输出搞不定。

Function Calling 让模型学会”调用工具”。用户说”查明天北京天气”，模型不会直接查天气，但输出一个结构化调用指令——{"function": "get_weather", "city": "北京", "date": "2026-05-27"}。外部系统收到指令，真的去调天气 API，把返回结果再喂给模型，模型组织语言回复。

Function Calling 是重要转折——从”会说话的机器”变成了”会用工具的助手”。

6. MCP——工具太多太乱，需要”USB 标准”

Function Calling 之后，新问题很快暴露：每个模型、每个应用都要单独对接工具，换模型就得重写。生态碎片化严重。

**MCP（Model Context Protocol）**是 Anthropic 在 2024 年底提出的标准化协议。定义了统一接口规范——大模型怎么发现工具有哪些能力、怎么调用、怎么拿到结果。开发者按这套协议写一个 MCP Server，任何支持 MCP 的 AI 应用都能直接用。

就好比 USB 之前每种设备有专属接口，MCP 是 AI 工具生态的”USB 标准”。

7. Agent——从”一问一答”到”自己干活”

前面的技术都在解决”单次问答”场景。但很多任务不是一次能搞定的。早期有个叫 AutoGPT 的项目，用 GPT-3.5 反复跑循环，当时挺轰动，现在回头看效果一般——但思路是对的。

Agent的核心是一个循环：思考 → 行动 → 观察 → 再思考 → ……

比如”调研竞品 X 的最新动态”——Agent 不是直接回答，而是自己规划：搜索关键词 → 筛选结果 → 信息不够 → 换关键词再搜 → 交叉验证 → 整理报告。每一步依赖上一步的输出。

本质很简单：Prompt + LLM + Tools = Agent。提示词给目标和约束，LLM 负责推理决策，Tools 负责执行。最简 Agent 就几十行代码——一个 while 循环，把工具返回的结果不断喂回模型。

8. Multi-Agent——一个人搞不定，组队

单 Agent 很快遇到瓶颈：任务太复杂、上下文太长、不同子任务需要不同能力。

Multi-Agent的思路：把任务拆了，不同 Agent 各管一摊——产品经理 Agent 梳理需求、开发 Agent 写代码、测试 Agent 找 Bug，各自有独立上下文和专长。像现在的 Claude Code、Codex 都内置了这种多 Agent 协作能力。

但这个方向有坑：多个 Agent 互相对话，Token 消耗暴涨；协作路径变复杂，出问题难定位。所以 只有在任务天然可拆分成独立子任务时才上 Multi-Agent，别为了多而多。

9. Context Engineering——上下文决定 Agent 上限

Agent 在循环中不断积累信息：对话历史、工具返回结果、中间推理步骤、背景知识……全得塞进 LLM 的上下文窗口。窗口有限，信息越多模型越可能”看漏”关键内容。

**Context Engineering（上下文工程）**就是解决这个的：怎么筛、怎么压、怎么排，让有限窗口装下最有价值的信息。比如对长对话做摘要压缩、按重要性排序插入上下文、用文件系统做”外部记忆”。

很多时候模型不是不够聪明，是”看到”的东西质量太差。上下文工程是 Agent 从”能用”到”好用”的关键。

10. Agent Skill——把能力打包，像装 App 一样

Agent 的能力越攒越多，Prompt 越来越长，维护越来越痛苦。而且很多能力是通用的——写了一个”代码审查”的 Agent，别人想复用得从头扒配置。

Agent Skills 是 Anthropic 提出的轻量级开放格式。把一整套 Agent 能力——Prompt 指令、工具脚本、知识文件——打包成可复用模块。Agent 运行时按需激活不同 Skill，用到哪个加载哪个（渐进式披露），不必把所有能力塞进 Prompt。

几个关键点：

  • Agent Skill 本质是子 Agent——装一个 Skill 等于配了一个专业小弟
  • 特别适合 SOP 沉淀——离职的同事终将化作温暖的 Skill，新人装上就用
  • agentskills.io 上已有大量社区 Skill

MCP 标准化了”工具怎么接”，Agent Skills 标准化了”能力怎么打包”，两者互补。

11. OpenClaw——Agent 不只在聊天框里

Agent 大多被绑在聊天界面里，主动去找它才能干活。但实际场景希望 Agent 能融入飞书、微信、邮件——像一个真正的助理。

OpenClaw 是开源的个人 AI Agent 框架，TypeScript 开发。核心创新是拓展了 Agent 的交互入口——不只是网页聊天，还能接入飞书等即时通讯工具。在飞书里发条消息”整理今天会议纪要”，Agent 就去干活，干完回来汇报。

精简版 nanobot 代码量更小，适合学习和二次开发。方向很清楚：从”你得打开聊天框找它”变成”它就在工作流里等着”。

12. Harness Engineering——给 Agent 套上缰绳

OpenAI 最近提出的概念。前面 11 个阶段都在解决”怎么让 AI 更强”，但 Harness Engineering 关心的是怎么让 AI 不乱来。

核心思想：给 Agent 构建受控环境——约束机制限制 Agent 能做什么不能做什么、反馈回路让 Agent 出错时自动纠偏、可观测性让每一步有迹可查。

把 Agent 比作员工：前面在教”怎么做”，Harness Engineering 在教”做对的事，不做错的事”。方向很清楚——AI 的下一个阶段不是更大更强，而是更可控、更可靠。

FAQ

Agent 和普通 chatbot 到底差在哪？

Chatbot 是”一问一答”，每次对话独立。Agent 会自己规划步骤、调用工具、根据中间结果调整策略。判断标准：任务需要 3 步以上完成，Chatbot 做不了，Agent 才有可能做。

该不该追 Multi-Agent？还是单 Agent 够了？

大部分场景单 Agent 足够。只有发现”上下文太长经常遗漏关键信息”或”任务天然需要不同专长角色协作”时才考虑 Multi-Agent。先做好一个 Agent，别为了多而多。

这些概念里，哪个对普通开发者影响最大？

MCP 和 Agent Skill。MCP 让工具一次开发到处用，不再绑死在一个模型上。Agent Skill 让经验和 SOP 沉淀成可复用模块。两个东西把 AI 开发从”手工作坊”变成了”可组装的工厂流水线”。
