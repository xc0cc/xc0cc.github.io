Git 解决了什么问题

写项目方案的时候经常出现这种情况：方案_v1.docx → 方案_v2.docx → 方案_最终版.docx → 方案_最终版2.docx。某天想找回 v1 里的一个段落，面对满屏文件完全无从下手。

这就是没有版本控制的日常：文件满天飞、改动不可追溯、协作全靠手动合并。

Git 做的事情很简单——记录文件的每一次变化。谁在什么时间改了什么，随时可以回溯。多人同时改同一个项目时，Git 自动合并各自的改动，冲突的地方标出来让人裁决。

Git 由 Linus Torvalds 在 2005 年为管理 Linux 内核开发而创建。到 2026 年，Stack Overflow 调查显示超过 94% 的开发者在使用 Git，已经是版本控制的事实标准。

核心概念：先理解，再操作

三个区域：工作目录 → 暂存区 → 本地仓库

Git 维护着三个逻辑区域：

类比拍照的流程：

  • 工作目录——摆 Pose，怎么摆都行
  • 暂存区（git add）——选好构图，告诉摄影师”就这个角度，准备拍”
  • 本地仓库（git commit）——按下快门，照片正式存进相册

分两步（add → commit）的设计是有原因的：改了五个文件，只想把其中三个放进这次提交。暂存区就是用来挑拣的。

分布式

Git 是分布式版本控制系统。git clone 下来的不只是最新文件，而是完整的仓库历史。这意味着没网也能 commit，回家再 push 同步。SVN 那种集中式的做不到——服务器挂了就没法提交。

克隆和同步

push 之前先 pull。如果远程有新提交，Git 会拒绝你的 push。先拉下来合并再推。

分支

分支是 Git 最重要的功能。可以理解为给项目复制了一份独立副本，在上面随便改，不影响主线。改满意就合并回去，改废了就删掉。

为什么需要分支

典型场景：在 main 分支修一个 Bug，修到一半突然需要紧急上线新功能。没有分支就卡住了——Bug 没修完不能提交，新功能又急着要。
